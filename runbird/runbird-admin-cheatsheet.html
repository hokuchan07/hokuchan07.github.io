基本原則

最初に、社長が知っておくべき大原則を1つだけ。

このGitHub管理画面、社長は基本的に触らなくて大丈夫です

RUNBIRDのナレッジAI運用は、すでに必要な設定が揃っている状態です。社長は普段はCursorで質問するだけ・人事/契約フォルダにメモが溜まっていくのを確認するだけ、で完結します。

とはいえ「これは何？」と気になった時のため、各画面の意味を一覧化しました。
操作が必要な時は、原則的に 鈴木にChatworkで一言 頂ければこちらで対応します。 誤操作で人事ナレッジが他の人に見える事故を防ぐためです。

画面別の意味と使い方

GitHub の各画面で、何ができて何がやっちゃダメかをまとめました。

●

① Organization トップ

github.com/runbird-inc

RUNBIRD全社のナレッジリポジトリ一覧が見えるトップページ。4つのナレッジ（福祉・IT・人事・契約書）がここに並んでいます。

できること

  • 各ナレッジリポジトリへ移動
  • 誰が何にアクセスできるか俯瞰
  • 新着Activity の確認

触らないこと

  • 「Settings」での Org rename
  • 「Delete this organization」
  • 有償プランへのアップグレード
●

② People（メンバー一覧）

github.com/orgs/runbird-inc/people

runbird-inc に所属している全メンバー（GitHubアカウント単位）が並ぶページ。誰がOrgに参加しているかの確認用。

できること

  • Org全メンバーの確認
  • 各メンバーのプロフィール確認
  • 新メンバーの招待（「Invite member」ボタン）

注意点

  • 招待するだけでは権限は付かない（Team追加が別途必要）
  • 招待後の Team 振り分けを忘れると見えるナレッジが定まらない
●

③ Teams（チーム一覧）— 権限グループ

github.com/orgs/runbird-inc/teams

8つのチーム（権限グループ）が並ぶページ。「誰がどのナレッジを見られるか」を決める仕組みの中核です。

できること

  • 各Teamのメンバー確認
  • Teamの権限（Read/Write/Admin）の確認

触らないこと

  • Team自体の削除
  • 権限レベルの変更
  • 新Team作成（領域追加時に鈴木が対応）
●

④ Team詳細（例：executives = 幹部）

github.com/orgs/runbird-inc/teams/executives

「幹部（executives）」チームの詳細画面。社長が見ていたのはここです。経営層3名（社長・鈴木・星崎）が登録されており、全ナレッジに Admin権限を持っています。

できること

  • 幹部メンバー一覧の確認
  • 新規幹部の追加（「メンバーを追加する」）
  • 子チーム（サブチーム）の作成

注意点

  • ここに追加する＝全ナレッジ（人事含む）が見える
  • 経営層以外を追加しないこと
  • 経営層追加は事前に鈴木とすり合わせ推奨
●

⑤ Repositories（リポジトリ一覧）

github.com/orgs/runbird-inc/repositories

4つのナレッジリポジトリ（welfare-knowledge / it-knowledge / hr-knowledge / contracts）の一覧画面。

できること

  • 各リポジトリの中身確認（ファイル閲覧）
  • 更新履歴の確認

触らないこと

  • リポジトリ削除
  • publicに変更（情報漏洩リスク）
  • Settings の Transfer / Archive

新メンバーを招待したい時（江口さんを例に）

新しい人をRUNBIRDナレッジに参加させる時の手順。社長が直接やる方法と、鈴木に依頼する方法の2つを示します。

推奨：鈴木にChatworkで依頼するのが一番安全＆速い

権限グループ（Team）の選択を間違えると、その方が見えてはいけないナレッジ（人事面談など）まで見えてしまいます。鈴木にお声がけ頂ければ、適切なTeamに振り分けて招待します。

Chatworkで送る一言の例：

「鈴木さん、新しい営業の○○さん（GitHub ID: ××）にIT営業ナレッジへのアクセスをお願いします」

→ 鈴木が30秒で対応します🙏
━━━━━━━━━━ または ━━━━━━━━━━

社長が直接招待する場合（江口さんの招待を例に）

1
事前準備：そのメンバーのGitHub IDを確認

例：江口さんの場合 → GitHub ID は TakuyaEguchi00
まだGitHub アカウントを持っていない人の場合は、先に作ってもらってIDを聞いてください。

2
People（メンバー一覧）にアクセス
https://github.com/orgs/runbird-inc/people
右上の「Invite member」ボタンをクリック
3
GitHub IDを入力して招待

検索欄に TakuyaEguchi00 を入力 → 候補から該当ユーザーを選択 →「Invite」

→ 江口さんに招待メールが送信されます。承認待ち状態に。

4
★最重要：適切なTeamに追加（権限を決める）

Org招待だけだと、その人は何にもアクセスできません。続いて Team に追加して権限を付与する必要があります。

例：江口さんを「IT営業 + 人事投入 + 契約書参照」にする場合

  1. github.com/orgs/runbird-inc/teams/it-write へ移動 → 「メンバーを追加する」→ TakuyaEguchi00 追加
  2. github.com/orgs/runbird-inc/teams/hr-input へ移動 → 同様に追加
  3. github.com/orgs/runbird-inc/teams/contracts-read へ移動 → 同様に追加

→ 3つのTeamに追加することで、3領域に対する適切な権限が付与されます。

5
本人にアクセス可能であることを伝える

招待メール承認 → 自動セットアップ実行 →Cursorで使用開始
運用ガイドも合わせて送ると親切：

その他の操作

招待以外でよくある操作も簡単にまとめています。

退職処理 重要

退職者が出た場合、Org からメンバー除外することで、全リポジトリのアクセスが一斉に剥奪されます。

1. People → 該当メンバーの「...」メニュー
2. 「Remove from organization」
3. 確認ダイアログで承認
→ 全ナレッジへのアクセス即時剥奪

異動：別のTeamに移動 通常

例：IT営業から福祉に異動した場合、it-writeから外して welfare-writeに追加。

1. 元Team（it-write）→ Removeでメンバー除外
2. 移動先Team（welfare-write）→ Add でメンバー追加
→ ナレッジアクセス権が切り替わる

権限を上げる：Read → Write

例：閲覧専門だった人に編集も任せたい場合、◯◯-read teamから◯◯-write team へ移動。

1. ◯◯-read team から Remove
2. ◯◯-write team に Add
→ 編集権限が付与される

新領域追加（例：動画事業ナレッジ）

新しいナレッジ領域が必要になった場合、新リポジトリ作成と新Team作成は鈴木が担当します。

ChatWorkで「動画事業のナレッジ作りたい」と一言
→ 鈴木が repo作成・Team作成・初期投入
→ 完成後、社長にお知らせ

困った時

i

判断に迷ったら、まず鈴木にご相談ください

権限管理は1度ミスをすると意図しない情報共有が起きるため、慎重に進めるのが原則です。
ChatWorkで「これって大丈夫？」と一言頂ければ、こちらで状況を確認・代行します🙏

!

絶対にやってはいけないこと

・Organizationの削除（「Delete this organization」）
・リポジトリのpublic化（情報漏洩）
・リポジトリ削除（GitHubから完全消去）
・有償プランへの「うっかり」アップグレード
この4つだけは避けてください。それ以外は基本的にリカバリ可能です。
